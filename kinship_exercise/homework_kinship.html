Problem statement

Create a program that can read in a dataset of individuals in a family and their parent and spouse connections, and determine how two people are related. You will define two classes: Person (representing a person from the family data) and Family (which will track all the people about whom you have relationship information).

You may optionally work with one classmate on this assignment. Please do not share your work with anyone other than that classmate.

Background

For this program we will be creating a simplified model of family relationships. Our model will include

  • the name of each person, which we will also use as a unique identifier

  • the gender of each person ('f' for female, 'm' for male, 'n' for nonbinary)

  • a person’s parents, if known

  • a person’s spouse, if applicable

In real life, individuals and families can be much more complex than this. If we were developing a relationship calculator for commercial purposes (or for most languages other than English), we would likely need to consider a number of other factors and use more complex algorithms.

Lowest common relative (LCR)

To determine the relationship between two individuals, we will try to identify the lowest common relative (LCR) of those individuals. Later on, we’ll provide an algorithm for finding the LCR of two people, but for now, we just want to introduce the concept and give a sense of what it means within our model of family relationships.

The LCR of two individuals could be one of the individuals themselves, or a third person that both individuals are connected to through a series of parent and/or spouse connections. A person is the LCR if no other person is more closely connected to both individuals through parent and/or spouse connections. Below are some examples of LCRs of various relationships.

  • The LCR of a mother and a daughter is the mother.

  • The LCR of two siblings is their parent (any shared parent will do).

  • The LCR of an aunt (say, the mother’s sister) and a nephew is the aunt’s parent, who is the nephew’s grandparent.

  • The LCR of two brothers-in-law (say, the husband of a woman and the brother of the woman) is a shared parent of the woman and her brother. (The woman is not the LCR because, while her husband has a direct spouse connection to her, but the brother has no direct connection to her in our model—​the connection between two siblings is a shared parent.)

In many cases there will be two or more LCRs—​in those cases it shouldn’t matter which one is chosen as the LCR for purposes of determining the relationship between the two people.

Relationship paths

An important step in determining how two people are related is to identify the path of parent and spouse connections that link each person to the lowest common relative. In this assignment, a relationship path will consist of a sequence of the letters "P" and "S", where "P" represents a parent connection and "S" represents a spouse connection. We will ignore all paths that involve more than one "S" connection. Here are some examples of relationship paths:

  • All people have a connection to themselves; the path to oneself is an empty string: "".

  • The path to a person’s parent is "P"; the path to a grandparent is "PP"; the path to a great-grandparent is "PPP", and so on.

  • The path to a person’s spouse is "S". The path to a spouse’s parent is "SP".

  • The path to a step-parent is "PS". Note that when there is more than one path from a person to a relative, we will always use the shortest path, so a biological parent will always have the path "P", not "SP", even if that parent is married to a person’s other parent.

Combined paths to a shared relative

When we have identified a shared relative for two people (Person1 and Person2), we can create a combined path from each of the two people to the shared relative. We will do this by concatenating Person1’s relationship path to the shared relative, a colon, and Person2’s relationship path to the shared relative ("PATH1:PATH2"). Once we’ve identified the complete set of shared relatives, whichever relative has the shortest combined path is the LCR; if combined paths of the same length exist for multiple relatives, any of them can be chosen as LCR.

The combined path to the LCR, coupled with the gender of Person1, will give us all the information we need to fill in the blank in the statement, "Person1 is Person2’s _". Below are some examples of combined paths and corresponding relationships:

  • The combined path :P indicates that Person1 is Person2’s parent (LCR is Person1).

  • The combined path P: indicates that Person1 is Person2’s child (LCR is Person2).

  • The combined path :PP indicates that Person1 is Person2’s grandparent (LCR is Person1).

  • The combined path P:P indicates that Person1 is Person2’s sibling (LCR is their parent).

  • The combined path PP:P indicates that Person1 is Person2’s niece/nephew/nibling (LCR is Person1’s grandparent/Person2’s parent).

  • Combined paths S: or :S indicate that Person1 is Person2’s spouse. (LCR is either Person1 or Person2; it doesn’t matter which.)

  • The combined path :SP indicates that Person1 is Person2’s parent-in-law (LRC is Person1).

Note that our program won’t have kinship terms for all possible combined paths. If we don’t have a term for a particular relationship, we will default to "distant relative".

If the combined path is present in relationships, its corresponding value will be a dictionary of kinship terms defining that combined path. If we look up Person1’s gender in the dictionary of kinship terms, we will retrieve the term that describes Person1’s relationship to Person2.

Person class

The Person class represents a person in a family. Instances of this class will have four attributes:

  • name: the person’s name (a string)

  • gender: the person’s gender (a string); should have one of the following values: 'f', 'm', 'n'

  • parents: the person’s parents, if known (a list of instances of Person; may be empty)

  • spouse: the person’s spouse, if applicable (an instance of Person, or None)

You will define the following methods: __init__(), add_parent(), set_spouse(), connections(), and relation_to(). All of these methods will have self as their first parameter.

__init__() method

The __init__() method should take a name and a gender as arguments. You can assume that both of these are strings and that the gender is one of the following values: 'f', 'm', 'n'.

This method should create a name attribute and a gender attribute to store the name and gender provided as arguments. It should also create an attribute called parents (set this equal to an empty list) and an attribute called spouse (set this equal to None).

add_parent() method

This method should take an instance of Person as an argument. It should append that instance to the parents attribute of self.

set_spouse() method

This method should take an instance of Person as an argument. It should store that instance as the value of the spouse attribute of self.

connections() method

This method takes no arguments. The purpose of this method is to identify connections to relatives who might be lowest common relatives of self and another instance of Person. The goal is to build a dictionary such that each key is an instance of Person to whom self is related via a combination of parent and/or spouse connections, and the corresponding value is a string indicating a path of connections from self to that instance of Person (see Relationship paths for more details). We will ignore paths that involve more than one spouse connection.

To build a dictionary of connections, we need to systematically follow parent and spouse connections of relatives of self until we have found all applicable relatives and determined the shortest path that connects them to self, bearing in mind the following rules:

  • There may be more than one path to a person. If we have already identified a path to that person, we won’t replace it.

  • If the path from self to a particular person includes a spouse connection, we won’t consider that person’s spouse.

In addition to building a dictionary of connections, we will build a queue to help us keep track of individuals whose parents and spouses we need to . A queue is a first-in, first-out data structure that operates like a line at the post office or a conveyor belt at the grocery store: new things get added to the back, and when we are ready to process something, we remove it from the front. We will use a list as our queue; we’ll use the append() method to add something, and pop(0) whenever we remove something (the 0 tells Python to remove the item at index 0).

Here is pseudocode for the algorithm we’ll use to find the connections of self:

  • Create a dictionary of connections (cdict) where self is a key with an empty string as its value

  • Create a queue of Person objects whose parent and spouse connections we need to follow; initially, the only thing in the queue should be self

  • While the queue is not empty:

    • Take the first Person object off the queue (we’ll call this person)

    • Look up the path from self to person in cdict; we will call this personpath

    • For each of person's parents:

      • If the parent is not in cdict:

        • The path to the parent is personpath plus a "P" at the end; add the parent as a new key in cdict with the path to the parent as the corresponding value

        • Add the parent to the end of the queue

    • If the value of personpath doesn’t contain "S" AND person has a spouse AND person's spouse isn’t in cdict:

      • The path to the spouse is personpath plus an "S" at the end; add the spouse as a new key in cdict with the path to the spouse as the corresponding value

      • Add the spouse to the end of the queue

  • Return cdict

Hints

Every key in cdict is an instance of Person who is a "connection" of self (in other words, a person who is connected to self in a way that makes them a potential LCR for some relationship). The value associated with a key in cdict is a string indicating how self is connected to that person. For example, if Elijah (located near O3 in the diagram) is self, at some point Brandon (located near M2 in the diagram) will end up in cdict with the path "P", because Brandon is Elijah’s parent. When we go to add Brandon’s parents to cdict--in other words, when the value of person is Brandon--personpath will be "P", because "P" is the value in cdict that goes with Brandon. "P" is the path from self (Elijah in our example) to Brandon. When we then go to add Brandon’s parents to cdict, the value we will associate with each of them is "PP"--personpath plus another "P".

So, when I say "Look up the path from self to person in cdict", what I mean is: look up person in cdict. The value you get back will be a string that is the path from self to person (for example, "P" if self is Elijah and person is Brandon).

relation_to() method

This method should take an instance of Person as an argument. If self and the other person are not related, it should return None. If there is a kinship term in relationships.relationships that describes self's relationship to the other person, this method should return that (see relationships.py module). If self and the other person are related but there is no kinship term in relationships.relationships that describes their relationship, this method should return "distant relative".

In order to determine the relationship between self and the other person, you will need to

  • call each object’s connections() method and store each dictionary of connections in its own variable

  • find the keys shared by the two dictionaries; if no keys are shared, return None

  • make an iterable of combined paths to each shared relative

  • find the shortest combined path in the iterable you just made (i.e., the path that consists of the fewest characters); this is the combined path to the LCR

  • if the combined path to the LCR is a key in relationships.relationships, use self.gender to look up the appropriate kinship term to describe how self is related to the other person; return that term

  • otherwise, return "distant relative"

Hints

  • There is a set operation that can help you find shared values from two containers. Just sayin'.

  • Once you have combined paths for all shared relatives, you are trying to find a minimum value from items in that set, using a custom criterion for what constitutes a minimum value. Does that sound like something we’ve talked about recently?

  • Once you know that the combined path to the LCR is a key in relationships.relationships, you can look up correct kinship term with one expression of the form DICT[OUTER_KEY][INNER_KEY]. DICT evaluates to a dictionary object. DICT[OUTER_KEY] then evaluates to the value associated with OUTER_KEY in that dictionary object, which happens to be another dictionary object. Finally, DICT[OUTER_KEY][INNER_KEY] evaluates to the value in the inner dictionary associated with the key INNER_KEY.

Family class

The Family class keeps track of all the Person instances you create. It will have the necessary machinery to create all the instances of Person and define parent and spouse relationships for those instances. It will also provide a convenient way for looking up the relationship between two people given their names.

Instances of this class will have one attribute, people. The value of this attribute will be a dictionary where each key is the name of a person and each value is a corresponding Person object.

You will define the following methods for this class: __init__() and relation(). Both of these methods will have self as their first parameter.

__init__() method

The __init__() method should take as an argument a dictionary with the following keys:

  • "individuals": a dictionary where each key is the name of a person and each value is that person’s gender ('f', 'm', or 'n').

  • "parents": a dictionary where each key is the name of a person and each value is a list of the names of the person’s parents. You can assume that every name in "parents" is also found in "individuals".

  • "couples": a list of lists; each inner list contains two names. These two people are married to each other. You can assume that every name in "couples" is also found in "individuals".

In this method, you will create an attribute of self called people; initially, it will be an empty dictionary. Then, do the following operations using the dictionary you received as an argument:

  • For each person in the "individuals" key, create an instance of Person with the name and gender of this person, and add the name and instance of Person to self.people.

  • For each person in the "parents" key:

    • Look up the Person object corresponding to that person.

    • For each of that person’s parents:

      • Look up the Person object corresponding to the parent.

      • Use the person’s add_parent() method to define a parent relationship between the person and their parent.

  • For each pair of people in the "couples" key:

    • Look up the Person objects corresponding to each person.

    • Use each person’s set_spouse() method to define a spouse relationship to the other person.

relation() method

This method should take the names of two people as arguments. We will assume that both names are keys in self.people.

The method should look up the Person objects corresponding to the two people and return the result of the first person’s relation_to() method, passing in the second person as an argument. The return value will either be None or a kinship term, expressed as a string; see relation_to() method for more details.
